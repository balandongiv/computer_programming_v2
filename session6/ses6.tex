\textbf{What is Object-Oriented Programming (OOP)?}
\begin{itemize}\setlength{\itemsep}{2pt}
    \item A \textbf{class} is a blueprint that describes what data and behavior a thing should have.

    \item A \textbf{method} is a function stored on the class that operates on the object.

\subsection{Motivation}
In Sessions 4 and 5, you built an Iris classifier using \textbf{functions}.
Your pipeline worked, but you may have noticed something repetitive: many functions needed the same configuration values (for example, the threshold, the positive label, and the negative label).
You solved this by passing a \texttt{settings} dictionary from function to function.

\subsubsection{Session 4}
In Session 4, your program used functions, but those functions depended on several global constants defined near the top of the file, such as \texttt{THRESHOLD}, \texttt{POSITIVE\_LABEL}, and \texttt{NEGATIVE\_LABEL}.
That means the function could work only because those names already existed somewhere else in the program.
For a beginner, this can feel convenient at first, because the function call looks short, but the function is actually relying on information stored outside itself.

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
THRESHOLD = 2.0
POSITIVE_LABEL = "setosa"
NEGATIVE_LABEL = "not_setosa"

def compute_threshold_prediction(sample):
    if sample["petal_length"] < THRESHOLD:
        return POSITIVE_LABEL
    return NEGATIVE_LABEL
\end{minted}

\noindent
When you read this function, you have to look somewhere else in the file to find out what \texttt{THRESHOLD} means.
So, even though the function has only one parameter, it is still using several extra pieces of information behind the scenes.

\subsubsection{Session 5}
In Session 5, you improved the design by starting to pass configuration values as arguments.
For example, \texttt{compute\_threshold\_prediction} now receives \texttt{threshold} directly, instead of reading the threshold from a global constant.
This is a step forward because the function becomes more explicit: you can see what information it needs just by looking at its parameter list.

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
def compute_threshold_prediction(sample, threshold):
    if sample["petal_length"] < threshold:
        return POSITIVE_LABEL
    return NEGATIVE_LABEL
\end{minted}

\noindent
However, as programs grow, several functions may all need the same shared configuration.
At that point, one common idea is to collect those related values into one dictionary, such as \texttt{settings}, and then pass that same dictionary into each function that needs it.

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
settings = {
    "threshold": 2.0,
    "positive_label": "setosa",
    "negative_label": "not_setosa",
}

def compute_threshold_prediction(sample, settings):
    if sample["petal_length"] < settings["threshold"]:
        return settings["positive_label"]
    return settings["negative_label"]

def derive_true_label(sample, settings):
    if sample["species"] == settings["positive_label"]:
        return settings["positive_label"]
    return settings["negative_label"]

def run_prediction_loop(dataset, settings):
    for sample in dataset:
        y_pred = compute_threshold_prediction(sample, settings)
        y_true = derive_true_label(sample, settings)
\end{minted}

\noindent
Here, passing a \texttt{settings} dictionary from function to function means that the same dictionary object is first passed into \texttt{run\_prediction\_loop}, and then that function passes it again into \texttt{compute\_threshold\_prediction} and \texttt{derive\_true\_label}.
In other words, the data keeps getting forwarded from one function call to the next so that every function can access the same configuration values.
This is clearer than relying on hidden global values, but it also creates repetition because many functions must accept and forward the same extra argument.

\subsection{Why Refactor Now}
Passing a \texttt{settings} dictionary is a useful step forward because it makes the shared configuration explicit.
However, for a first small program, what feels simple at first can become awkward as the program grows.
The main issue is not the dictionary itself.
The main issue is that the same configuration must be created, passed around, and kept consistent in many places.

\subsubsection{Problem 1: Many Classifiers Mean Many Dictionaries}
Suppose you want to compare several classifiers instead of only one.
For example, maybe one classifier uses threshold \texttt{1.8}, another uses \texttt{2.0}, and another uses \texttt{2.2}.
If your design depends on a \texttt{settings} dictionary, then each classifier needs its own dictionary.

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
settings_a = {
    "threshold": 1.8,
    "positive_label": "setosa",
    "negative_label": "not_setosa",
}

settings_b = {
    "threshold": 2.0,
    "positive_label": "setosa",
    "negative_label": "not_setosa",
}

settings_c = {
    "threshold": 2.2,
    "positive_label": "setosa",
    "negative_label": "not_setosa",
}
\end{minted}

\noindent
This is still manageable for two or three classifiers.
But if you want 10 classifiers with 10 thresholds, you now need 10 separate dictionaries that are almost identical.
That means you start writing the same information again and again.
Repeated code is harder to maintain because one small change may need to be copied into many places.
For example, if you decide to rename one key or change one default label, you may need to edit every dictionary.

\subsubsection{Problem 2: Every Function Call Must Carry the Correct Settings}
Once you store configuration in a dictionary, every function that needs that configuration must receive it.
So the program has to keep passing the same object from one function call to the next.

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
def run_prediction_loop(dataset, settings):
    for sample in dataset:
        y_pred = compute_threshold_prediction(sample, settings)
        y_true = derive_true_label(sample, settings)
\end{minted}

\noindent
For a beginner, the key idea is this: the code must always remember which \texttt{settings} dictionary belongs to which classifier.
If you accidentally pass \texttt{settings\_a} into one function and \texttt{settings\_b} into another, your program may mix two different configurations in a confusing way.
So each function call is doing two jobs at once:
it is running the function, and it is also carrying the correct configuration object along with it.

\subsubsection{Problem 3: New Settings Create More Edits Everywhere}
Now imagine that later you decide your classifier should also control whether each prediction is printed.
You might add a new setting called \texttt{print\_each}.
At first, this sounds like a very small change.
But in a function-based design, one new piece of shared configuration often spreads through many parts of the program.

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
settings = {
    "threshold": 2.0,
    "positive_label": "setosa",
    "negative_label": "not_setosa",
    "print_each": True,
}

def run_prediction_loop(dataset, settings):
    for sample in dataset:
        y_pred = compute_threshold_prediction(sample, settings)
        y_true = derive_true_label(sample, settings)
        if settings["print_each"]:
            print(sample["id"], y_pred)
\end{minted}

\noindent
This version still works, but now more parts of the program depend on \texttt{settings}.
If several functions need the new value, then several function bodies must be updated.
If you were not using a dictionary yet, then you would also need to change multiple function signatures, such as turning \texttt{compute\_threshold\_prediction(sample, threshold)} into a longer function call with more arguments.
In a larger program, this creates a chain reaction:
one new setting can force edits in many function definitions and many call sites.

\noindent
In Session 6 we solve this by using a \textbf{class}.
The core idea is:
\begin{quote}
\textbf{Bundle the configuration (data) and the behavior (functions) into one place.}
\end{quote}

\noindent
\textbf{Refactoring promise:} the classifier rule itself stays the same.
This session reorganizes where the code lives, but it does \textbf{not} change what ``setosa'' means or how the threshold rule works.

\section{The Blueprint: Defining a Class}

A class is like a \textbf{blueprint} for building objects.
Think of a cookie cutter: the cutter itself is the class, and every cookie you make from it is one object (or ``instance'').

\medskip
\noindent
The class definition in Python looks like this:

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
class IrisRuleClassifier:
    # Everything indented here belongs to the class
    pass
\end{minted}

\noindent
The class name uses \textbf{CamelCase} by convention (each word starts with a capital letter).

\subsection{What is \texttt{self}?}

Every method inside a class receives a special first argument called \texttt{self}.
\textbf{\texttt{self} means ``this specific object''.}

\medskip
\noindent
When you write:

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
self.threshold = 2.0
\end{minted}

\noindent
you are storing the value \texttt{2.0} \textbf{inside this particular object}.
A different object can store a different value at the same attribute name without interfering.

\medskip
\noindent
You do \textbf{not} pass \texttt{self} yourself when calling a method.
Python passes it automatically.
When you write \texttt{classifier.predict(sample)}, Python actually calls
\texttt{IrisRuleClassifier.predict(classifier, sample)} behind the scenes.

\begin{tcolorbox}[title=Task 1: Define the \_\_init\_\_ Method]
The special method \texttt{\_\_init\_\_} (two underscores on each side) runs \textbf{automatically} whenever you create a new object.

\section{Methods: Functions That Belong to an Object}

A \textbf{method} is simply a function that lives inside a class.
The only real difference from a regular function is:
\begin{itemize}\setlength{\itemsep}{2pt}
    \item a regular function stands alone and has no built-in access to shared data,
    \item a method belongs to an object and can read or update the object's data through \texttt{self}.
\end{itemize}

\noindent
Even if a method does not use any attribute (like the simple status printer below), it still receives \texttt{self} as its first parameter because it belongs to the class.

\subsection{Example: Refactoring a Function into a Method}

Before asking you to refactor the prediction methods yourself, let's see one complete example of how to convert a standalone function into a method.

\medskip
\noindent
In Session 5, you may have used a standalone function like \texttt{make\_print\_status} that printed status messages.
In Session 6, we convert it into a class method called \texttt{print\_status}.

\medskip
\noindent
\textbf{The standalone function version (Session 5 style):}
\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
def make_print_status(status_text):
    print(f"[STATUS] {status_text}")
\end{minted}

\noindent
\textbf{The class method version (Session 6 style):}
\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
class IrisRuleClassifier:
    # ... __init__ above ...

    def print_status(self, status_text):
        """Print a status message."""
        print(f"[STATUS] {status_text}")
\end{minted}

\noindent
\textbf{What changed?}
\begin{enumerate}\setlength{\itemsep}{2pt}
    \item The function is now \textbf{indented inside the class}.
    \item It receives \texttt{self} as the first parameter (even though it does not use any attributes).
    \item You call it using \texttt{classifier.print\_status("Build dataset")} instead of \texttt{make\_print\_status("Build dataset")}.
\end{enumerate}

\noindent
\textbf{How to call it:}
\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
classifier = IrisRuleClassifier()
classifier.print_status("Build dataset")
classifier.print_status("Run prediction loop")
\end{minted}

\noindent
This reads naturally: ``ask this classifier object to print a status message''.

\section{The Evaluation Methods}

The next three methods form the evaluation pipeline.

\subsection{Task 4: Implement update\_result\_counts}

\begin{tcolorbox}[title=Task 4: Implement update\_result\_counts]
After predicting and finding the true label for one sample, we need to update our running totals.
This method takes the current counters as inputs, updates them, and returns them.

\medskip
\noindent
\textbf{Why return values instead of just changing something?}
Because beginners find explicit return values easier to follow than hidden state changes.
When you return the updated values, you can immediately see what changed.

\medskip
\noindent
In your template, find the comment labelled \textbf{Task 4}.
Complete \texttt{update\_result\_counts(self, correct, wrong, total, y\_pred\_list, y\_pred, y\_true)} so that it:
\begin{enumerate}
    \item Compares \texttt{y\_pred} and \texttt{y\_true}.
    \item If they match, increments \texttt{correct} by 1; otherwise increments \texttt{wrong} by 1.
    \item Always increments \texttt{total} by 1 (even when the prediction is wrong).
    \item Appends \texttt{y\_pred} to \texttt{y\_pred\_list}.
    \item Returns the tuple \texttt{(correct, wrong, total, y\_pred\_list)}.
\end{enumerate}
\end{tcolorbox}

\noindent
The return tuple lets the caller unpack the updated values cleanly:

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
correct, wrong, total, y_pred_list = classifier.update_result_counts(
    correct, wrong, total, y_pred_list, y_pred, y_true
)
\end{minted}

\begin{tcolorbox}[title=Checkpoint]
\textbf{Critical self-check:} does \texttt{total} increase by 1 for every sample, even when the prediction is wrong?
If you only increment \texttt{total} when the prediction is correct, your accuracy will always report 100\% --- which is wrong!
\end{tcolorbox}

\begin{tcolorbox}[title=Common Mistake]
Placing \texttt{total += 1} inside the \texttt{if} block.
It must be outside and after both branches so it runs no matter what.
\end{tcolorbox}

\subsection{Task 6: Implement run\_prediction\_loop}

\begin{tcolorbox}[title=Task 6: Implement run\_prediction\_loop]
This is the ``orchestrator'' method.
It loops through every sample in the dataset, calls the other methods for each sample, and returns the final evaluation counters.

\medskip
\noindent
\textbf{What you will see here} is the main benefit of grouping methods in a class:
one method can call other methods on the same object using \texttt{self.method\_name(...)}.

\medskip
\noindent
In your template, find the comment labelled \textbf{Task 6}.
Complete \texttt{run\_prediction\_loop(self, dataset)} so that it:
\begin{enumerate}
    \item Starts \texttt{correct}, \texttt{wrong}, \texttt{total} at \texttt{0}.
    \item Starts \texttt{y\_pred\_list} as an empty list \texttt{[]}.
    \item Prints a header line: \texttt{"\textbackslash n=== Start Session 6 Prediction Loop ==="}.
    \item Loops through each \texttt{sample} in \texttt{dataset}:
    \begin{enumerate}
        \item Calls \texttt{self.compute\_threshold\_prediction(sample)} to get \texttt{y\_pred}.
        \item Calls \texttt{self.derive\_true\_label(sample)} to get \texttt{y\_true}.
        \item Calls \texttt{self.update\_result\_counts(...)} and unpacks the returned tuple.
        \item Prints one trace line per sample (see format below).
    \end{enumerate}
    \item Returns the tuple \texttt{(correct, wrong, total, y\_pred\_list)}.
\end{enumerate}

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}

)
\end{minted}
\end{tcolorbox}

This method is the clearest example in Session 6 of methods calling each other.
\texttt{run\_prediction\_loop} does not contain the prediction logic itself --- it \textbf{delegates} to \texttt{compute\_threshold\_prediction} and \texttt{derive\_true\_label}.
That is good design: each method stays focused on its own single job.

\begin{tcolorbox}[title=Task 8: Use the Class in main()]
Now that all the methods are complete, connect them inside \texttt{main()}.

\medskip
\noindent
In your template, find the comment labelled \textbf{Task 8}.
Complete the \texttt{main()} function so that it:
\begin{enumerate}
    \item Calls \texttt{setup\_application\_list()} to load the dataset.
    \item Creates a \texttt{classifier = IrisRuleClassifier(threshold=2.0)}.
    \item Calls \texttt{classifier.print\_status("Build dataset")}.
    \item Calls \texttt{classifier.print\_status("Run prediction loop")}.
    \item Calls \texttt{classifier.run\_prediction\_loop(dataset)} and unpacks the returned tuple.
    \item Calls \texttt{classifier.calculate\_accuracy(correct, total)} to get \texttt{accuracy}.
    \item Calls \texttt{classifier.print\_status("Print summary")}.
    \item Calls \texttt{classifier.print\_summary(correct, wrong, total, y\_pred\_list, accuracy)}.
\end{enumerate}
\end{tcolorbox}

\noindent
When your \texttt{main()} is complete, calling the classifier should look clean and readable:

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
def main():
    dataset = setup_application_list()

    classifier = IrisRuleClassifier(threshold=2.0)

    classifier.print_status("Build dataset")
    classifier.print_status("Run prediction loop")

    correct, wrong, total, y_pred_list = classifier.run_prediction_loop(dataset)
    accuracy = classifier.calculate_accuracy(correct, total)

    classifier.print_status("Print summary")
    classifier.print_summary(correct, wrong, total, y_pred_list, accuracy)
\end{minted}

Notice that you no longer pass a \texttt{settings} dictionary anywhere.
The threshold and labels live inside the \texttt{classifier} object.

\section{Review and Next Steps}

\textbf{Summary:}
\begin{itemize}
    \item \textbf{Classes} group data (attributes) and behavior (methods) together in one place.
    \item \textbf{Objects} remember their own configuration, which means each object can operate independently.
    \item \textbf{\texttt{self}} is how a method accesses the object's own attributes and other methods.
    \item \textbf{\texttt{\_\_init\_\_}} is the constructor: it runs automatically when an object is created and sets up starting data.
    \item Methods can call each other inside the same class using \texttt{self.method\_name(...)}.
    \item Object-oriented design is the foundation of major machine learning libraries like \texttt{scikit-learn}.
\end{itemize}

\textbf{Review Questions:}
\begin{enumerate}
    \item What does \texttt{self} refer to inside a method?
    \item What is the difference between \texttt{threshold} and \texttt{self.threshold} inside a method?
    \item Why did we give \texttt{derive\_true\_label} only one job (convert a species name to a lesson label)?
    \item Why does \texttt{total += 1} need to be outside the \texttt{if/else} block in \texttt{update\_result\_counts}?
    \item How would you create a second classifier with a different threshold without changing the first one?
\end{enumerate}

\textbf{Micro-Exercise:} Create two classifier objects with different thresholds and compare their accuracies on the same dataset.
For example, try \texttt{threshold=1.4} and \texttt{threshold=5.0}.
Which one performs best?

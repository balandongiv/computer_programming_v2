\begin{enumerate}

But what happens when you want to try a \textbf{second type} of classifier --- say, one that measures distances instead of comparing to a threshold?

\medskip
\noindent
With the Session 6 design, you would need to copy and paste the entire evaluation loop into every new classifier class.
That creates \textbf{duplication}.

\subsection{Why Duplication Is a Problem}

\begin{itemize}\setlength{\itemsep}{2pt}
    \item If you discover a bug in the evaluation logic, you must fix it in every copy.
    \item If you improve the loop, you must update every class.
    \item Adding a third or fourth classifier type becomes repetitive work.
\end{itemize}

\noindent
The solution is \textbf{inheritance}: write the shared evaluation logic once in a \textbf{parent class}, and let each classifier type be a \textbf{child class} that only writes its own prediction rule.

\begin{tcolorbox}[title=Real-World Hook]
Inheritance is like defining a general concept of ``Vehicle'' (it moves, it has wheels) and then creating specific child concepts like ``Car'' and ``Bicycle'' that inherit the general behavior but add their own specific rules (engine vs.\ pedals). Similarly, our classifiers share evaluation behavior but differ in how they predict.
\end{tcolorbox}

\section{The Parent Class (\texttt{ClassifierBase})}

A parent class (also called a \textbf{base class}) is a more general blueprint.
It contains code that many child classes can reuse.

\medskip
\noindent
The key insight for this lesson is:
\begin{quote}
The base class does not know \textbf{how} a specific classifier predicts a label.\\
But it \textbf{does} know how to test any classifier on a dataset.
\end{quote}

\noindent
So the parent class will contain:
\begin{itemize}
    \item \texttt{derive\_true\_label} --- convert a raw species name to a lesson label
    \item \texttt{update\_result\_counts} --- update counters after each sample
    \item \texttt{calculate\_accuracy} --- compute accuracy from counters
    \item \texttt{evaluate} --- the shared loop that calls all of the above
\end{itemize}

\noindent
And the parent class will \textbf{not} contain a working \texttt{predict} method, because prediction logic is different for every child class.

\subsection{The Starter: ClassifierBase and predict with NotImplementedError}

The parent class definition starts like this:

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
class ClassifierBase:
    def __init__(self):
        self.positive_label = "setosa"
        self.negative_label = "not_setosa"

    def predict(self, sample):
        """Child classes must provide their own version of this method."""
        raise NotImplementedError("Child class must implement predict()")
\end{minted}

\noindent
\textbf{What is \texttt{raise NotImplementedError}?}

\medskip
\noindent
Think of it as a placeholder that says:
\begin{quote}
``I know this method needs to exist, but I am refusing to guess how it should work.
Every child class must write its own version.''
\end{quote}

\noindent
If you create a child class and forget to write \texttt{predict}, Python will raise this error and remind you.
This is much more helpful than getting a mysterious wrong answer.

\medskip
\noindent
The \texttt{\_\_init\_\_} method of \texttt{ClassifierBase} stores the two lesson labels.
These labels will be inherited by all child classes automatically, so every child classifier already knows the positive and negative label without redefining them.

\begin{tcolorbox}[title=Task 4: Implement evaluate]
This is the most important method in the parent class.
It is the \textbf{shared evaluation loop} that every child classifier will inherit and use without rewriting.

\medskip
\noindent
\textbf{How it works:} the loop calls \texttt{self.predict(sample)} on each sample.
Because \texttt{predict} is not defined in the parent, it will call the child class's version instead.
This is the mechanism that makes inheritance useful.

\medskip
\noindent
In your template, find the comment labelled \textbf{Task 4}.
Complete \texttt{evaluate(self, dataset)} so that it:
\begin{enumerate}
    \item Starts \texttt{correct}, \texttt{wrong}, \texttt{total} at \texttt{0}.
    \item Starts \texttt{y\_pred\_list} as an empty list.
    \item Prints a header line: \texttt{"\textbackslash n=== Start Evaluation ==="}.
    \item Loops through each \texttt{sample} in \texttt{dataset}:
    \begin{enumerate}
        \item Calls \texttt{self.predict(sample)} to get \texttt{y\_pred}.
        \item Calls \texttt{self.derive\_true\_label(sample)} to get \texttt{y\_true}.
        \item Calls \texttt{self.update\_result\_counts(...)} and unpacks the returned tuple.
        \item Prints one trace line per sample (same format as Session 6).
    \end{enumerate}
    \item After the loop, calls \texttt{self.calculate\_accuracy(correct, total)}.
    \item Returns the tuple \texttt{(correct, wrong, total, y\_pred\_list, accuracy)}.
\end{enumerate}

\medskip
\noindent

)
\end{minted}
\end{tcolorbox}

\noindent
\textbf{The magic line in this method is:}

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}

\end{minted}

\noindent
When you call \texttt{evaluate} on a \texttt{RuleClassifier} object, Python calls \texttt{RuleClassifier.predict(sample)}.
When you call it on a \texttt{NearestCentroidClassifier} object, Python calls \texttt{NearestCentroidClassifier.predict(sample)}.
The parent does not need to know which child it is.
That is the power of inheritance.

\section{The First Child Class (\texttt{RuleClassifier})}

Now we create the first child class.
This child class will inherit everything from \texttt{ClassifierBase} and only needs to provide its own \texttt{predict} method.

\subsection{Inheritance Syntax}

To make one class inherit from another, write the parent class name in
parentheses after the child class name:

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
class RuleClassifier(ClassifierBase):
    pass
\end{minted}

\noindent
This single line says: ``\texttt{RuleClassifier} is a \texttt{ClassifierBase}.
It inherits all of its methods and attributes.''

\subsection{Calling the Parent \texttt{\_\_init\_\_} with \texttt{super()}}

When a child class has its own \texttt{\_\_init\_\_}, it must also run the parent's \texttt{\_\_init\_\_} first.
You do this with \texttt{super().\_\_init\_\_()}.

\medskip
\noindent
\textbf{What does \texttt{super()} mean?}

\medskip
\noindent
\texttt{super()} refers to the parent class.
So \texttt{super().\_\_init\_\_()} means:
\begin{quote}
``Run the parent class's \texttt{\_\_init\_\_} method first, then proceed with my own setup.''
\end{quote}

\noindent
In practice, the parent's \texttt{\_\_init\_\_} sets up \texttt{self.positive\_label} and \texttt{self.negative\_label}.
If you forget to call \texttt{super().\_\_init\_\_()}, those attributes will not exist when the child methods try to use them.

\section{The Second Child Class (\texttt{NearestCentroidClassifier})}

Now we create a completely different type of classifier.
Instead of using a fixed threshold, this classifier uses a distance calculation.

\medskip
\noindent
\textbf{The idea:} there are two ``centre points'' (centroids) for petal length:
\begin{itemize}
    \item a typical setosa petal length (e.g., \texttt{1.5} cm)
    \item a typical non-setosa petal length (e.g., \texttt{4.5} cm)
\end{itemize}

\noindent
For each sample, we compare the sample's petal length to both centre points and choose the closer one.
The label of the closer centre becomes the prediction.

\medskip
\noindent
\textbf{What changes compared to RuleClassifier?}
Only \texttt{predict} is different.
The evaluation loop, accuracy calculation, and label conversion are exactly the same.
That is why we keep them in the parent class.

\subsection{Task 8: Implement NearestCentroidClassifier.predict}

\begin{tcolorbox}[title=Task 8: Implement NearestCentroidClassifier.predict]

\medskip
\noindent
In your template, find the comment labelled \textbf{Task 8}.
Complete \texttt{predict(self, sample)} inside \texttt{NearestCentroidClassifier} so that it:
\begin{enumerate}
    \item Reads \texttt{petal\_length = sample["petal\_length"]}.
    \item Computes \texttt{dist\_to\_setosa = abs(petal\_length - self.setosa\_center)}.
    \item Computes \texttt{dist\_to\_not\_setosa = abs(petal\_length - self.not\_setosa\_center)}.
    \item If \texttt{dist\_to\_setosa < dist\_to\_not\_setosa}, returns \texttt{self.positive\_label}.
    \item Otherwise, returns \texttt{self.negative\_label}.
\end{enumerate}
\end{tcolorbox}

\noindent
The built-in function \texttt{abs(x)} returns the absolute value (the distance from zero) of a number.
For example, \texttt{abs(-3.1)} returns \texttt{3.1}.
This is how we compute a ``distance'' between two numbers without caring about direction.

\section{Comparing Both Classifiers}

Now we have two classifier objects that use different prediction rules but share the same evaluation interface.

\begin{minted}[fontsize=\small, linenos, frame=lines]{python}
rule_clf = RuleClassifier(threshold=2.0)
centroid_clf = NearestCentroidClassifier(setosa_center=1.5, not_setosa_center=4.5)

print("\nRuleClassifier results:")

print(f"Accuracy: {round(accuracy, 2)}%")

print("\nNearestCentroidClassifier results:")
correct, wrong, total, y_pred_list, accuracy = centroid_clf.evaluate(dataset)
print(f"Accuracy: {round(accuracy, 2)}%")
\end{minted}

\noindent
Notice:
\begin{itemize}\setlength{\itemsep}{2pt}
    \item \texttt{rule\_clf.predict(sample)} uses the threshold rule.
    \item \texttt{centroid\_clf.predict(sample)} uses the distance rule.
    \item \texttt{rule\_clf.evaluate(dataset)} and \texttt{centroid\_clf.evaluate(dataset)} use the \textbf{same} shared code.
\end{itemize}

\noindent
This is the key insight: evaluation is shared, prediction is specific.

\section{Polymorphism}

Now that you have two classifiers with the same interface, we can introduce one of the most important ideas in object-oriented programming: \textbf{polymorphism}.

\medskip
\noindent
\textbf{Polymorphism} means:
\begin{quote}
Different objects can be used in the same way if they share the same method names.
\end{quote}

\noindent
In our case:
\begin{itemize}
    \item \texttt{RuleClassifier} has a method called \texttt{evaluate}.
    \item \texttt{NearestCentroidClassifier} also has a method called \texttt{evaluate}.
\end{itemize}

\noindent
So code can call \texttt{.evaluate(dataset)} on either object without needing to know which specific classifier type it holds.
The code does not care whether the object is a \texttt{RuleClassifier} or a \texttt{NearestCentroidClassifier}.
It only cares that the object supports \texttt{evaluate}.

\medskip
\noindent
This is exactly how scientific libraries like \texttt{scikit-learn} are designed.
Every model in scikit-learn has a \texttt{fit} method and a \texttt{predict} method.
You can swap in a different model without changing the code that calls \texttt{fit} and \texttt{predict}.

\section{Review and Next Steps}

\textbf{Summary:}
\begin{itemize}
    \item \textbf{Inheritance} lets a child class reuse code from a parent class without copying and pasting it.
    \item \textbf{Parent class} holds shared behavior (evaluation pipeline).
    \item \textbf{Child classes} only write what is different (prediction rule).
    \item \textbf{\texttt{super().\_\_init\_\_()}} runs the parent's setup before the child adds its own attributes.
    \item \textbf{\texttt{raise NotImplementedError}} is a clear signal that a method must be provided by the child.
    \item \textbf{Polymorphism} means different objects can be used through the same interface if they share the same method names.
\end{itemize}

\textbf{Review Questions:}
\begin{enumerate}
    \item What is the difference between a parent class and a child class?
    \item Why does calling \texttt{evaluate()} on a child object run the child's \texttt{predict()} and not the parent's?
    \item What happens if you forget to call \texttt{super().\_\_init\_\_()} inside a child's \texttt{\_\_init\_\_}?
    \item Why is \texttt{run\_and\_report} a good polymorphism example?
    \item What is one advantage of having \texttt{evaluate} in the parent class instead of copying it into each child?
\end{enumerate}
